## 🎯 Цель
Создать **один, полностью структурированный markdown‑файл**, который:
1. Описывает текущее состояние BIM‑платформы (технологический стек, архитектура, структура репозитория).
2. Перечисляет **критические проблемы** (баги, уязвимости, узкие места производительности) с указанием уровня тяжести.
3. Предлагает **пошаговый план** по превращению прототипа в готовый к публичному использованию сервис, размещённый в Яндекс‑Облаке (или Beget).
4. Содержит практические рекомендации по **документации, SEO и CI/CD**.
5. Детализирует **разработку нового функционала**, описывает необходимые экраны и UI‑компоненты.
6. Описывает процесс **миграции проекта на GitLab** и настройки CI/CD в GitLab.

## 🔍 Текущее состояние
- **API**: Полностью реализованы CRUD‑операции для пользователей и проектов, Swagger UI доступен по `/swagger`.
- **Аутентификация**: JWT‑based, но секретный ключ хранится в `appsettings.json` (нужно вынести в переменные окружения).
- **CORS**: Разрешён только `http://localhost:3000`; для публичного сайта потребуется добавить домен.
- **IFC‑viewer**: Интегрирован через `IFC.js`/`Three.js` на фронтенде, но нет прогрессивной загрузки и fallback‑решения.
- **Работа с файлами**: Есть базовый эндпоинт загрузки, но нет сканирования на вирусы и версионирования.
- **Документация**: README покрывает установку, но отсутствуют схемы архитектуры, гайд по деплою и полная API‑спецификация.
- **Тесты**: Юнит‑ и интеграционные тесты не написаны ни для бэкенда, ни для фронтенда.
- **CI/CD**: Нет GitHub Actions или иных пайплайнов.
- **Дизайн**: UI использует Tailwind, но нет тёмной темы, микровзаимодействий и фирменного брендинга.

## ⚠️ Критические ошибки и риски
| Серьёзность | Проблема | Где обнаружено | Последствия | Предлагаемое решение |
|--------------|----------|----------------|-------------|----------------------|
| **Критическая** | Секрет JWT хранится в открытом виде | `Program.cs` (строка 44) | При утечке репозитория компрометируются все токены | Переместить секрет в переменные окружения или в Яндекс Secrets Manager |
| **Критическая** | SQLite не подходит для многопользовательского продакшн | `Program.cs` (строка 12) | Потеря данных, невозможность масштабирования | Перейти на PostgreSQL или MySQL, добавить миграции EF Core |
| **Высокая** | CORS ограничен localhost | `Program.cs` (строки 18‑28) | Публичный сайт будет блокироваться браузером | Добавить список разрешённых доменов через переменные окружения |
| **Высокая** | Отсутствует валидация загружаемых файлов | `src/app/services/fileService.ts` (предположительно) | Возможна загрузка вредоносных файлов, DoS‑атаки | Внедрить серверную проверку (ClamAV), ограничить размер и типы файлов |
| **Средняя** | Нет ролевой модели доступа к проектам | API‑контроллеры | Неавторизованный доступ к данным | Реализовать политики ASP.NET Core (Admin, Contractor, Viewer) |
| **Средняя** | Отсутствие автоматических тестов | Весь репозиторий | Риск регрессий, низкое качество кода | Добавить xUnit‑тесты для бэкенда, Jest + React Testing Library для фронтенда |
| **Низкая** | UI не адаптирован под планшеты | `src/app/components/*` | Плохой UX на мобильных устройствах | Использовать responsive‑утилиты Tailwind, протестировать на разных экранах |
| **Низкая** | Нет SEO‑мета‑тегов и Open Graph | `pages/_document.tsx` | Низкая видимость в поиске | Добавить `<Head>` с title, description, OG‑тегами |

## 🚀 Дорожная карта (MVP → Публичный релиз)
### Фаза 1 – Стабилизация (2‑3 недели)
1. **Укрепление безопасности**
   - Перенести JWT‑секрет и строки подключения в переменные окружения.
   - Добавить ролевую авторизацию.
   - Реализовать серверную валидацию файлов и сканирование на вирусы.
2. **Миграция БД**
   - Ввести PostgreSQL через Docker‑Compose.
   - Написать миграции EF Core.
3. **База тестов**
   - Добавить xUnit‑тесты для ключевых сервисов.
   - Добавить Jest + React Testing Library для UI‑компонентов.
4. **CI/CD**
   - Настроить GitHub Actions: lint → test → build → Docker‑image.
5. **Обновление документации**
   - Диаграмма архитектуры (PlantUML), гайд по деплою, спецификация API.

### Фаза 2 – Улучшение IFC‑viewer (2 недели)
- Интегрировать готовую библиотеку **`web-ifc-viewer`** для быстрой загрузки, разрезов и свойства элементов.
- Добавить lazy‑loading больших IFC‑файлов.
- Предусмотреть fallback‑UI для неподдерживаемых браузеров.
- **Новый функционал**: возможность рисовать карандашом непосредственно в 3D‑вьюере, привязывать рисунок к координате модели и сохранять как часть комментария.
- При клике на такой комментарий камера автоматически перемещается к сохранённой позиции, отображается рисунок в виде наложения.
- Реализовать хранение аннотаций (координаты камеры, вектор направления, SVG‑рисунок) в базе данных и синхронизацию в реальном времени через SignalR.
- Добавить UI‑компоненты: `AnnotationCanvas` (canvas поверх 3D‑вьюера), `CommentWithSketch` (форма комментария с мини‑превью), кнопки «Рисовать», «Сохранить», «Отменить».

### Фаза 3 – Публичный хостинг и масштабирование (3 недели)

- Включить автоскейлинг и CDN для статических ресурсов.
- Добавить Redis для кэширования метаданных моделей.

### Фаза 4 – Коллаборация и полировка UI (2‑3 недели)
- Реальное время комментариев к элементам модели через **SignalR**.
- Редизайн UI: тёмная тема, glass‑morphism карточки, микровзаимодействия (Tailwind `animate-` утилиты).
- Внедрить **русскую локализацию** (i18next с ru‑языком).
- Провести аудит производительности с Lighthouse, оптимизировать размер бандла.

### Фаза 5 – Релиз и мониторинг (1 неделя)
- Запуск в продакшн, включить мониторинг (Yandex Cloud Metrics, Sentry).
- Опубликовать публичный SLA и канал поддержки.
- Вывести `PROJECT_ANALYSIS.md` и `PROJECT_OVERVIEW.md` в репозиторий GitHub.

## 🛠️ Разработка нового функционала и UI‑экраны
### 1. Управление проектами (Project Management)
- **Экран «Список проектов»** – таблица/карточки с фильтрами, поиск, пагинацией.
- **Экран «Создание/Редактирование проекта»** – форма с полями: название, описание, клиент, статус, даты.
- **Экран «Детали проекта»** – боковая панель с информацией, список IFC‑моделей, кнопки «Загрузить файл», «Просмотр», «Комментарии».
- **API**: `GET /api/projects`, `POST /api/projects`, `PUT /api/projects/{id}`, `DELETE /api/projects/{id}`.
- **Компоненты**: `ProjectList`, `ProjectCard`, `ProjectForm`, `ProjectDetail`.

### 2. IFC‑viewer с расширенными возможностями
- **Экран «Просмотр модели»** – 3D‑вьюер, панель слоёв, свойства выбранного элемента, инструменты измерения.
- **Функции**: переключение видимости слоёв, секционирование, поиск элементов, экспорт выбранных данных.
- **Компоненты**: `IfcViewer`, `LayerControl`, `PropertyPanel`, `MeasurementTool`.

### 3. Управление файлами (File Management)
- **Экран «Загрузка файлов»** – drag‑&‑drop зона, прогресс‑бар, валидация типа и размера.
- **Экран «История версий»** – список версий файла, возможность отката.
- **API**: `POST /api/files`, `GET /api/files/{id}`, `GET /api/files/{id}/versions`, `DELETE /api/files/{id}`.
- **Компоненты**: `FileUpload`, `FileList`, `FileVersionList`.

### 4. Пользователи и роли (User & Role Management)
- **Экран «Список пользователей»** – таблица с ролями, статусом, действиями (изменить роль, блокировать).
- **Экран «Профиль пользователя»** – личные данные, настройки, смена пароля.
- **API**: `GET /api/users`, `POST /api/users`, `PUT /api/users/{id}`, `DELETE /api/users/{id}`.
- **Компоненты**: `UserList`, `UserCard`, `UserProfile`.

### 5. Комментарии и коллаборация (Collaboration)
- **Экран «Комментарии к модели»** – список комментариев, привязка к элементу модели, возможность отвечать.
- **API**: `GET /api/comments`, `POST /api/comments`, `PUT /api/comments/{id}`, `DELETE /api/comments/{id}`.
- **Компоненты**: `CommentThread`, `CommentForm`, `CommentItem`.

### 6. Административная панель (Admin Dashboard)
- **Экран «Статистика»** – графики использования, количество проектов, активные пользователи.
- **Экран «Настройки системы»** – управление переменными окружения, секретами, интеграциями.
- **Компоненты**: `DashboardStats`, `SystemSettings`.

#### План разработки UI‑экрана
1. **Прототипирование** – создать wireframes в Figma (или аналог) для каждого экрана.

## 🌐 Миграция проекта на GitLab
### 1. Подготовка репозитория
- Создать новый проект в GitLab (Private или Internal).
- Добавить `README.md`, `.gitignore` (уже есть) и `LICENSE`.
- Выполнить `git remote add gitlab <gitlab‑url>` и `git push -u gitlab --all`.

### 2. Перенос CI/CD
- Создать файл `.gitlab-ci.yml` в корне репозитория.
- Определить стадии: `lint`, `test`, `build`, `docker`, `deploy`.
- Пример базового конфига:
```yaml
stages:
  - lint
  - test
  - build
  - docker
  - deploy

lint_back:
  stage: lint
  image: mcr.microsoft.com/dotnet/sdk:8.0
  script:
    - dotnet format --verify-no-changes

lint_front:
  stage: lint
  image: node:20-alpine
  script:
    - npm ci
    - npm run lint

test_back:
  stage: test
  image: mcr.microsoft.com/dotnet/sdk:8.0
  script:
    - dotnet test --no-build --verbosity normal

test_front:
  stage: test
  image: node:20-alpine
  script:
    - npm ci
    - npm test

build_back:
  stage: build
  image: mcr.microsoft.com/dotnet/sdk:8.0
  script:
    - dotnet publish -c Release -o out

build_front:
  stage: build
  image: node:20-alpine
  script:
    - npm ci
    - npm run build

docker_image:
  stage: docker
  image: docker:latest
  services:
    - docker:dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $CI_REGISTRY_IMAGE:latest .
    - docker push $CI_REGISTRY_IMAGE:latest

deploy:
  stage: deploy
  image: alpine:latest
  script:
    - echo "Deploy to Yandex Cloud (or Beget) using kubectl/ssh"
    # Здесь добавить команды деплоя, например, kubectl apply -f k8s/
  only:
    - main
```
- Добавить переменные CI/CD в Settings → CI/CD → Variables (например, `JWT_SECRET`, `DB_CONNECTION_STRING`, `YANDEX_CLOUD_TOKEN`).

### 3. Настройка GitLab‑раннеров
- Установить GitLab Runner на сервер (можно использовать shared runners).
- Зарегистрировать runner с тегами `docker`, `dotnet`, `node`.
- Убедиться, что runner имеет доступ к Docker‑демону для сборки образов.

### 4. Перенос Issue‑tracker и Wiki
- При необходимости экспортировать задачи из GitHub (если были) и импортировать в GitLab через CSV.
- Перенести Wiki‑страницы (можно скопировать markdown‑файлы в репозиторий `docs/`).

### 5. Обновление ссылок и документации
- Заменить все ссылки в `README.md` и других файлах с `github.com` на `gitlab.com`.
- Обновить инструкцию по развёртыванию, указав команды `git push gitlab` и запуск пайплайна.
